# Design QA Agent - 项目总结

---

## 一句话说明

> **让产品、测试、设计师无需懂代码，也能发现样式问题并清晰告诉开发"哪里要改"**

---

## 核心理念

**设计系统 Tokens 是真理源（Single Source of Truth），而非每张设计稿。**

- 设计稿可以滞后，不影响开发规范
- 只要代码正确使用 Tokens，样式一定正确
- 减少设计师和开发之间的沟通成本

---

## 解决什么问题

| 痛点 | 解决方案 |
|------|----------|
| 设计稿和开发有差异 | 可视化标注差异 |
| 人工走查效率低 | 系统化检查流程 |
| 需求方说不清问题 | 截图 + 标注 + 描述 |
| 开发缺少设计品味 | 工具辅助自测 |
| 没有完整设计稿 | 以 Tokens 规范为准 |

---

## 目标用户

- **产品经理**：验收交付质量
- **测试人员**：样式走查测试
- **设计师**：设计还原验收
- **前端开发**：自测自查

---

## 核心流程

```
上传截图 → 标注问题 → 指派开发 → 开发修复 → 验收关闭 → 完成 ✅
```

---

## 产品形态

**浏览器插件**
- 贴近使用场景，浏览器里直接用
- 轻量级，不需要额外打开应用
- 可截图当前页面

---

## MVP 功能范围

### ✅ 第一版做
- 上传截图
- 手动圈选标注
- 问题列表管理
- 指派给开发
- 状态流转（待处理 → 已解决）
- 导出问题报告

### ⏳ 后续迭代
- URL 自动截图
- AI 自动检测差异
- 飞书/Linear 自动同步
- Tokens 规范自动校验

---

## 产品路线图

| 阶段 | 目标 | 主要功能 |
|------|------|----------|
| **Phase 1** | MVP 验证 | 插件 + 截图 + 手动标注 + 问题管理 |
| **Phase 2** | 智能化 | AI 检测 + Tokens 校验 + 修复建议 |
| **Phase 3** | 协作化 | 飞书集成 + 质量看板 + CI/CD |

---

## 价值主张

| 传统方式 | Design QA Agent |
|----------|-----------------|
| 设计稿是标准 | Tokens 是标准 |
| 肉眼对比，效率低 | 工具辅助，系统化 |
| 问题说不清楚 | 截图标注，清晰明确 |
| 沟通来回多次 | 闭环流程，一次搞定 |
