## 一、项目背景

### 痛点
- 设计稿和开发产物之间经常存在样式差异（间距、颜色、字号等）
- 人工走查效率低、容易遗漏
- 设计师和开发之间的沟通成本高

### 目标用户 ⭐

**核心用户：需求方（非技术人员）**

| 角色 | 特点 | 痛点 |
|------|------|------|
| **产品经理** | 不懂代码 | 看不出样式问题，只能说"感觉不对" |
| **测试人员** | 不懂代码 | 只能肉眼对比，效率低、易漏 |
| **设计师** | 懂设计不懂代码 | 验收时发现问题，但说不清怎么改 |

**次要用户：开发人员**
- 能看懂代码 ✅
- 但缺少设计品味 ❌（发现不了"差了2px"、"颜色偏了"）
- 需要一个工具帮他们自测

**核心洞察**
> 开发人员可以看懂代码，但需求方看不懂。
> 这个工具让非技术人员也能发现样式问题，并**清晰地告诉开发"哪里要改"**。

---

## 二、核心功能

### 功能 1：设计稿导出 & 开发
- 从 Figma 获取设计稿信息（通过 API 或 MCP）
- 自动生成前端代码（HTML/CSS/React/Vue）
- 支持设计系统 tokens 的映射

### 功能 2：样式对比检测
- 输入：Figma 设计稿 + 开发页面 URL
- 输出：差异报告
  - 可视化标注（叠加对比、差异高亮）
  - 具体差异列表（哪个元素、什么属性、差多少）
  - 修复建议（甚至自动生成修复代码）

## 七、用户场景深挖

### 现实情况
- 很多小需求**没有设计稿**或设计稿不完善
- 开发有很多**自由发挥**的时刻
- 但自由发挥也需要**遵守设计系统规范**

### 两大场景

| 场景 | 检测目标 | 频率 |
|------|----------|------|
| **A：有完整设计稿** | 还原是否一致（像素级对比） | 较少 |
| **B：无/不完整设计稿** | 是否正确使用 tokens（规范检查） | 更常见 |

### 场景 B 的核心问题

```
❌ 硬编码写法：
   color: #1160fe;
   padding: 17px;
   font-size: 15px;

✅ 规范写法：
   color: var(--brand-default);
   padding: var(--spacing-4);
   font-size: var(--text-body);
```

即使视觉上"看起来对"，但如果用了硬编码：
- 主题切换会出问题
- 设计系统更新时不会同步
- 维护成本高

### 产品新定位

**Design System Linter** - 设计系统规范检查器

- 不只是"对比差异"
- 更是"检测规范遵守度"

### 核心理念 ⭐

> **设计系统 tokens 是真理源（Single Source of Truth），而不是每一张设计稿。**

设计稿可能没有更新，但只要代码正确使用了 tokens：
- 样式一定符合规范
- 主题切换自动适配
- 设计系统更新时全局生效

### 检测逻辑

```
代码：color: #1161fe
        ↓
匹配：这是 tokens 里 brand-default 的值
        ↓
判断：硬编码 ❌ 还是 var(--brand-default) ✅
        ↓
输出：警告 + 修复建议 + 一键替换
```

### 价值主张

| 传统方式 | 新方式（本工具） |
|----------|-----------------|
| 设计稿是标准 | tokens 是标准 |
| 设计稿没更新 = 无法验收 | 只要用对 tokens = 样式正确 |
| 每个页面逐像素对比 | 检测是否使用规范变量 |
| 设计师改稿 → 开发改代码 | 更新 tokens → 全局自动生效 |

## 八、协作流程设计（Checkflow）

### 核心理念
> 不只是检测工具，而是一个**闭环的样式走查工作流**

### 流程图

```
发起检查 → 问题列表 → 分配修复 → 验收关闭 → 完成 ✅
```

### 详细步骤

| 阶段 | 操作者 | 动作 |
|------|--------|------|
| 1️⃣ 发起检查 | 设计师/PM/QA | 输入设计稿 + 开发页面，触发检测 |
| 2️⃣ 问题列表 | 系统 | 自动生成差异，支持手动补充标注 |
| 3️⃣ 分配修复 | 发起人 | 指派问题给开发，开发标记"已修复" |
| 4️⃣ 验收关闭 | 发起人 | 确认问题解决，关闭或重新打开 |
| 5️⃣ 完成 | 系统 | 所有问题关闭 = 检查通过，生成报告 |

### 协作功能

- **多人协作**：团队成员都能查看、评论
- **标注高亮**：在截图上圈出问题位置
- **问题状态**：Open → In Progress → Resolved → Closed
- **@提人**：通知相关开发
- **历史记录**：追踪修改过程

### 参考产品

| 产品 | 借鉴点 |
|------|--------|
| Figma 评论 | 标注交互、实时协作 |
| GitHub PR Review | 问题状态管理、Resolve 机制 |
| Jira | 工作流、指派、优先级 |
| Linear | 简洁的 Issue 管理 |

## 九、交互 & 输出设计

### 输入方式
- **URL**：输入开发页面地址，自动截图检测
- **截图**：直接上传截图进行对比

### 输出结果
- **过程中**：问题列表 + 标注 + 协作讨论
- **最终产物**：问题解决日志/报告
  - 记录所有问题及解决过程
  - 可归档、可追溯
  - 作为质量验收凭证

### 问题分级

**设计原则**：简单为主，灵活可调

| 设置 | 说明 |
|------|------|
| 默认 | 无等级，统一进入"待处理"池 |
| 手动调整 | 可标记为高优先级 |
| 优先面板 | 高优问题单独归集，优先处理 |

### 问题面板结构

```
🔴 高优先级 (N)    ← 手动标记的紧急问题
⚪ 待处理 (N)      ← 默认问题池
✅ 已解决 (N)      ← 已关闭的问题
```

### 集成平台
- **飞书**：国内团队
- **Linear**：海外/效率优先团队

### Issue 结构设计

```
📋 [大 Issue] 首页样式走查 - 2026.02.10    ← 项目立项
│
├── 🔴 子 Issue: 按钮颜色不符合规范 → @开发A
├── 🔴 子 Issue: 标题间距差 8px → @开发A
├── ⚪ 子 Issue: 卡片圆角不一致 → @开发B
├── ⚪ 子 Issue: 阴影缺失 → @开发B
└── ...
```

### 自动化流程

| 步骤 | 动作 |
|------|------|
| 1. 检测到问题 | 系统自动创建子 Issue |
| 2. 指派开发 | 子 Issue 关联到具体开发 |
| 3. 开发修复 | 开发完成后关闭子 Issue |
| 4. 自动汇总 | 所有子 Issue 关闭 → 大 Issue 自动完成 |

### 通知触发点
- 新问题创建时 → 通知被指派的开发
- 问题被标记高优 → 通知相关人
- 问题被关闭 → 通知发起人验收
- 所有问题完成 → 通知团队

### 核心原则

```

### 功能范围

| 功能 | ✅ MVP 包含 | ❌ 后续迭代 |
|------|------------|-----------|
| 输入 | 上传截图 | URL 自动截图 |
| 标注 | 手动圈选 + 文字描述 | AI 自动检测差异 |
| 协作 | 创建问题、指派人 | 飞书/Linear 自动同步 |
| 状态 | 待处理 → 已解决 | 优先级面板 |
| 输出 | 问题列表 | 完整报告导出 |
| 形态 | 浏览器插件 | Web 应用 |

### 插件界面草图

```
┌─────────────────────────────────┐
│ 🔍 Design QA                  ✕ │
├─────────────────────────────────┤
│ [截图当前页面]  [上传设计稿]     │
├─────────────────────────────────┤
│ 📍 点击页面标注问题              │
├─────────────────────────────────┤
│ 问题列表 (3)                    │
│ • 按钮颜色 → @开发A    [✓]      │
│ • 间距问题 → 未指派    [ ]      │
│ • 字体大小 → @开发B    [ ]      │
├─────────────────────────────────┤
│ [导出报告]                      │
└─────────────────────────────────┘

### 选择理由

| 决策 | 原因 |
|------|------|
| 先截图，后 URL | 截图简单，不用处理网页兼容问题 |
| 先手动，后 AI | 先验证流程，AI 是锦上添花 |
| 先基础状态 | 简单够用，复杂分级后续加 |
| 浏览器插件 | 贴近使用场景，随手就能用 |

## 十五、补充决策 & 待讨论

### 已明确

| 项目 | 决策 |
|------|------|
| **产品形态** | 插件 或 Agent（待定） |
| **对比颗粒度** | 从组件级做起，不是整页 |
| **输入限制** | 限制截图大小 / 组件数量，从小块开始 |
| **像素精度** | 像素级差异不允许有误差 |
| **问题合并** | 类似问题自动发现并合并 |
| **核心场景** | 规范检查（而非设计稿还原） |
| **商业模式** | 暂不考虑 |

### 对比策略

```
从小到大，逐步修正：

单个组件 → 组件组合 → 模块 → 页面
   ↓           ↓         ↓       ↓
  精确       合并相似    汇总    报告
```

### 问题合并逻辑

```
发现 5 个按钮都有颜色问题
        ↓
合并为 1 个问题：「按钮颜色不符合规范」
        ↓
关联所有相关位置
        ↓
修复一次，全部解决
```
